# exocmd__Grounding_serviceCallPayload

`exo__StringProperty` on `exocmd__Grounding` (cardinality 0..1).

Opaque JSON config payload for `service_call` groundings. YAML serialization: `serviceCallPayload: '{"property":"..."}'` or `serviceCallPayload: '{"prototype":"$target","instanceClass":"ems__Task","folder":"03 Knowledge/inbox"}'`.

## Substitution semantics

Before `JSON.parse`, the value is run through `GroundingExecutor.substituteVariables` with the target IRI and `userInput`. Supported tokens: `$target`, `$now`, `$today`, `$nowLocal`, `$targetStart`, `$input`, `$value`. Already-parsed defaults merge into `userInput` for the registered service.

## Per-service payload structure (informal)

- `createAsset` accepts `{prototype, instanceClass, folder}`.
- `updateProperty` accepts `{property}`.
- Class-flip dispatch (Convert to Task / Project) uses `exocmd__Grounding_targetValueRef` instead — the value IS a class reference, not a service payload.

## Disjointness

Disjoint with `exocmd__Grounding_appendExpression` (b6f51a69-8310-48db-aedc-63865e2a8c66). A grounding MUST NOT carry both payload predicates simultaneously.

## Rationale

See RFC 918a2b65-a19f-4c38-bbce-86d3e58400bb. Selected over Phase 2 trio reuse because service_call JSON payload is multi-key and structurally unrelated to property_set's ref/literal/substitution trichotomy.
