# exocmd__Grounding_prefillLabelWithDate

`exo__DatatypeProperty` on `exocmd__Grounding` (cardinality 0..1, range `xsd:boolean`).

Опт-ин флаг для grounding'ов типа `create_instance`. Когда `true` — модал, открывающийся перед созданием инстанса, предзаполняет поле `label` значением `${prototype.exo__Asset_label} YYYY-MM-DD` (например, `Morning Wim Hof 2026-05-17`).

## Background

Восстанавливает поведение плагина v15.38 (`CreateInstanceCommand.showModal()` → `showLabelInputModal(defaultLabel)`), утраченное после миграции команд на vault-driven exocmd pipeline. В commit `abdb19a3` (PR #2733, 2026-04-11) при парсинге `exocmd__Grounding_inputSchema` маппинг полей в `CommandResolver` сократился до `name/type/label/required` и потерял `default`/`defaultValue` — а сам JSON Schema не умеет шаблонизировать `${parent.label} ${today}` динамически.

Hardcode prefill'а в плагине вместо template engine'а в RDF — осознанное решение (см. PR description). Альтернативы (template engine, динамические JSON Schema поля) расширяют scope сверх regression-fix'а и попадают в отдельный RFC.

## Semantics

- **`true`** — `CommandExecutionFlow` перед открытием модала проверяет inputSchema на наличие поля `name: "label"` без `defaultValue` и подставляет `${grounding.prototypeLabel} ${YYYY-MM-DD}`. Дата вычисляется в момент клика — кеширование `CommandResolver` не делает её «вчерашней».
- **`false` / отсутствует** — поведение неизменно (модал открывается с пустым полем `label`).
- **`defaultValue` уже задан в JSON Schema** — user-explicit override побеждает prefill.
- **Прототип не разрешён** (нет `targetPrototype` или нет `exo__Asset_label` у прототипа) — graceful fallback на пустое поле, ошибки не возникает.

Применяется **только** к `Grounding_type=create_instance`; на других типах флаг игнорируется.
