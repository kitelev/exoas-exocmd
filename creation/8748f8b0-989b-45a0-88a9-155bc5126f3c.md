# Create project

Grounding that creates a new `ems__Project` with automatic parent-context inheritance via the declarative `create_instance` pipeline.

RFC v2 Phase 4a (Issue #3165, 2026-05-23): migrated from `service_call`/`serviceId: createRelatedProject` to declarative `create_instance`. The legacy handler `ServiceRegistryPopulator.createRelatedProject` (lines 458-520) wrote: (a) folder = parent file's folder → expressed via `targetFolder: $targetFolder`; (b) status = Draft → PropertyDefault `d9aa9bb8`; (c) area-vs-parent inheritance → InheritanceRules `3f08f5a8` (cond Area, prio 100) + `43731bae` (excl Area, prio 50); (d) isDefinedBy inheritance → InheritanceRule `cbe000c4`.

**Plugin requirement (legacy)**: v15.92.0+ for `createRelatedProject` handler. Post-Phase 4b, the legacy handler is removed and this declarative form is the only path.

**Behavioral edge case (deferred to Phase 4b validation):** default `linkBackProperty = exo__Asset_prototype` is written unconditionally after the InheritanceRule step. Same caveat as sibling `4919afb1`.

**History**: this Grounding previously called the generic `createAsset` handler with `{"prototype":"ems__ProjectPrototype"}` (which never inherited parent context); it was switched to `createRelatedProject` (RFC-028 Finding 2). Phase 4a unifies both forks onto the single declarative pipeline.
