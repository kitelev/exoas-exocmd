# Create Class grounding

`exocmd__Grounding` (type `create_instance`) for the homoiconic «Create Class»
button (project 85150d63, W3).

The button is bound to the `exo__Ontology` metaclass, so the click target is
always an ontology-definition page. This grounding:

- `targetClass = [[8619c4fc-...]]` (the `exo__Class` **metaclass**) — the new
  asset's `exo__Instance_class` (it IS an `exo__Class`), via the Universal
  Default Template's `exo__Instance_class = $grounding.targetClass`
  PropertyDefault. (Inverts T1 Create Instance, where the host is the class and
  no `targetClass` is declared.)
- `targetFolder = $isDefinedByFolder` — co-locates the new class in the folder of
  its `exo__Asset_isDefinedBy` ontology (= the host ontology, see PropertyDefault
  below), resolved at execution time by the injected `RefToFolderResolver`.
  Because `isDefinedBy === host`, co-location also holds via the host-folder
  fallback when no resolver is wired.
- `inputSchema` — a one-field form: `label` (text). No ontology picker: the new
  class is defined by the HOST ontology, not a user-chosen one.
- `propertyDefault` → sets `exo__Asset_isDefinedBy` to the host's own UID via the
  `$targetClassSelf` token (`257a0a81-...`); the host IS the ontology.

uid / createdAt / label / aliases come from the Universal Default Template
(`label = $userInputLabel`).
