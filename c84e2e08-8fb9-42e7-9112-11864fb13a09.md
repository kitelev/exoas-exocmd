# Create note

Grounding action that calls the `createAsset` service to create a new `ztlk__FleetingNote`.

Uses the `service_call` grounding type with `inputSchema` to collect the note title from the user,
`targetValue` for the prototype + folder defaults, and the standalone `exocmd__Grounding_isDefinedBy`
wikilink to pin the owner identity (UID `0aa339bc-9b56-400a-8148-cbde57bbf0b6` = `a.kitelev`).

The owner is authored as a real frontmatter wikilink (not embedded in JSON) so the `a.kitelev`
asset's layout / Asset Relations panel lists every Grounding that pins it. Precedence in
`createAsset`: explicit `userInput.isDefinedBy` (from this property, merged by `GroundingExecutor`)
wins over parent inheritance and the registrar-injected vault default.

`targetValue.folder` is the literal default inbox path (`03 Knowledge/inbox`) used when the command
fires from the global Obsidian Command Palette (no active file → no parent folder to inherit). For
the inline-button surface (rendered on a `ztlk__FleetingNotePrototype` asset), the active file's
parent folder still wins — `userInput.folder` is honoured by the `createAsset` service handler
*before* the targetIRI-based inference branch, but the literal here doesn't override an explicit
button-side userInput. See `ServiceRegistryPopulator.ts:createAsset` for the resolution chain
and RFC `1429fcd0` PR-1 for the contract change that made literal folder safe.
