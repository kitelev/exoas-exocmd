# Create related task via service

Grounding that creates a new `ems__Task` related to the current asset (target's folder, area-vs-parent inheritance applied automatically).

RFC v2 Phase 4a (Issue #3165, 2026-05-23): migrated from `service_call`/`serviceId: createRelatedTask` to declarative `create_instance`. The legacy handler `ServiceRegistryPopulator.createRelatedTask` performed: (a) folder = parent file's folder → expressed via `targetFolder: $targetFolder`; (b) status = Draft → PropertyDefault `d9aa9bb8`; (c) area-vs-parent inheritance → InheritanceRule `3f08f5a8` (cond Area, prio 100) + `43731bae` (excl Area, prio 50); (d) isDefinedBy inheritance → InheritanceRule `cbe000c4` (prio 10).

**Behavioral edge case (deferred to Phase 4b validation):** default `linkBackProperty = exo__Asset_prototype` overwrites the area/parent slot only on Area-target invocation; the InheritanceRule writes `ems__Effort_area` (kept), then back-link unconditionally writes `exo__Asset_prototype: [[<Area>]]` (semantically odd but harmless). Phase 4b soak window will surface any user-visible regression.
