# exocmd__Grounding_appendExpression

`exo__StringProperty` on `exocmd__Grounding` (cardinality 0..1).

Value-expression for `property_append` groundings. YAML serialization: `appendExpression: "$target.exo__Asset_label"` (or any substitution expression supported by `substituteVariables`).

## Substitution semantics

The expression is evaluated by `GroundingExecutor.substituteVariables`. In addition to the standard tokens (`$target`, `$now`, `$today`, `$nowLocal`, `$targetStart`, `$input`, `$value`), property-accessor postfix is supported: `$target.<frontmatter_property>` reads the target asset's named frontmatter field. The resolved string is then appended set-wise (deduplicated) to the grounding's `exocmd__Grounding_targetProperty` on the target.

## Cardinality at executor level

Effective cardinality is `1` for `property_append` groundings — the executor fails-loud if neither `appendExpression` nor legacy `targetValue` is set. SHACL-lite enforces conditional cardinality (if `grounding_type = property_append` then this predicate MUST be present); the property definition itself remains `0..1` because it does not apply to other grounding types.

## Disjointness

Disjoint with `exocmd__Grounding_serviceCallPayload` (aaca11ba-48ab-4f22-bf64-7cb0f136b69d). A grounding MUST NOT carry both payload predicates simultaneously.

## Rationale

See RFC 918a2b65-a19f-4c38-bbce-86d3e58400bb. Phase 2 trio's `targetValueSubstitution` (range = closed-world `exocmd__SubstitutionToken` enum) cannot model property-accessor postfix; a new predicate keeps the expression syntax explicit and SubstitutionToken's enum closed.
